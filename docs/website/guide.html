Quick Start

The fastest way to run OpenDataMask is with Docker Compose:

git clone https://github.com/MaximumTrainer/OpenDataMask.git
cd OpenDataMask

# Generate secrets
export JWT_SECRET=$(openssl rand -base64 32)
export ENCRYPTION_KEY=$(openssl rand -base64 32 | head -c 32)

# Start all services
docker-compose up -d

# Open the UI
open http://localhost
Default ports — Frontend: 80, Backend API: 8080, PostgreSQL: 5432.

Step-by-Step Setup Guide

The following walkthrough covers the full lifecycle of a masking project — from creating an account through to a completed masking job with custom PII detection rules.

Screenshots are generated automatically from a live environment by verification/take_screenshots.py. Run pip install playwright && playwright install chromium then python3 verification/take_screenshots.py to regenerate them.
1

Register & Sign In

Navigate to http://localhost to reach the login page. Click Register to create your first account, or sign in with existing credentials. All passwords are bcrypt-hashed; credentials never leave your infrastructure.

2

Create a Workspace

A Workspace is the top-level container for a masking project. It holds all your database connections, table configurations, data mappings, and job history. Click New Workspace, enter a name and optional description, then click Create.

04-workspaces-list.png

The workspace dashboard — lists all your workspaces. Click + New Workspace to create your first project.

The New Workspace modal — give your project a name and optional description.

07-workspace-overview.png

The workspace overview tab — shows a summary of connections, tables, and recent job activity.

Use descriptive workspace names like "Prod → QA Sync" or "GDPR Compliance Prep" to make multi-project setups easy to navigate.
3

Connect Your Databases

Navigate to the Connections tab inside your workspace. You need at least two connections: a source (production-like data) and a destination (target environment for masked data). Supported types include PostgreSQL, MySQL, Azure SQL, MongoDB, and file uploads.

For each connection, fill in:

  • Name — a label used in job configuration
  • Type — database engine (e.g., POSTGRESQL)
  • Host / Port / Database — connection coordinates
  • Username / Password — stored encrypted with AES-256
  • Role — tick Source or Destination (or both)

09-add-connection-modal.png

The Add Connection form — enter the host, database name, credentials, and role.

Filling in the source connection — enable the Source role toggle and click Add Connection.

4

Configure Tables & Column Generators

The Tables tab lets you tell OpenDataMask which tables to process and how to transform each column. For every table, choose a masking mode (MASK, GENERATE, PASSTHROUGH, or SKIP), then configure a generator for each column that should be anonymised.

11-tables-configured.png

The Tables tab listing configured tables — click + Add Table to add a new one.

13-add-table-modal.png

Configuring the users table — selecting MASK mode.

12-generators-expanded.png

Column generators configured: full_name → FULL_NAME, email → EMAIL, phone → PHONE. Each mapped column will receive realistic synthetic data when the job runs.

Use Run Sensitivity Scan from the workspace Actions tab to auto-detect PII columns and receive generator recommendations — then click Apply Recommendations to configure them in one shot.
5

Data Mappings (Advanced Column Control)

The Data Mappings tab gives you fine-grained, per-column control that overrides the table-level generator configuration. Use it to set each column to one of three actions:

  • Mask — replace the value with a generated fake (choose strategy: FAKE, HASH, NULL, REDACT, PARTIAL_MASK, or REGEX)
  • Migrate As-Is — copy the original value unchanged
  • Omit — exclude the column entirely from the output

The wizard walks you through three steps: select a connection → select a table → configure each column.

14-data-mapping-wizard.png

Step 1 — Choose the connection whose schema you want to configure.

15-data-mapping-columns.png

Step 2 & 3 — Select the table, then set the action and masking strategy for each discovered column. Changes are saved immediately.

6

Run a Masking Job

With connections and table configuration in place, go to the Jobs tab and click Run New Job. Select your source and destination connections, then click Run. The job runs asynchronously — the progress badge cycles from PENDING → RUNNING → COMPLETED (or FAILED) in real time.

16-jobs-list.png

The Jobs tab — lists all past and running jobs. Click Run New Job to start a masking run.

17-run-job-modal.png

The Run New Job modal — select the source and destination connections, then click Run.

19-job-logs.png

Expanding the job log shows per-table row counts and any warnings raised during masking.

After a successful job, connect to your destination database to verify. The verification/verify.py script automates this check — it validates that PII columns contain synthetic data, primary keys are preserved, and no original values leaked through.
7

Custom PII Detection Rules

OpenDataMask ships with built-in sensitivity detection for common PII (email addresses, phone numbers, SSNs, etc.). The Sensitivity Rules page under Settings lets you define your own detection rules for domain-specific identifiers — for example, an internal employee ID format or a proprietary customer reference number.

Each custom rule has:

  • Name — displayed in scan reports and data mapping labels
  • Matchers — column-name patterns and/or regex patterns that identify the PII type
  • Data type filter — optionally restrict the rule to text, numeric, or date columns
  • Linked preset — optionally auto-apply a masking strategy when the rule fires
20-sensitivity-rules.png

The Sensitivity Rules settings page — lists all built-in and custom PII detection rules.

21-new-pii-rule-drawer.png

The New Rule side panel — give the rule a name and add one or more column-name matchers.

22-pii-rule-configured.png

A custom rule targeting columns named employee_id — the next sensitivity scan will flag matching columns and recommend a masking strategy.

After saving a new sensitivity rule, run a fresh Sensitivity Scan from the workspace Actions tab to see it take effect.

npm run dev                          # dev server (port 5173)
npm run build                        # production build

CLI (Go)

cd cli
go build -o opendatamask-cli ./...

Workspaces

A Workspace is the top-level scope for a masking project. It contains connections, table configurations, jobs, data mappings, and custom PII rules. Workspaces can inherit configuration from a parent workspace for multi-environment setups.

Masking Modes

ModeDescription
MASKReplace column values with generated fake data using the configured generator
GENERATEGenerate an entirely new row set regardless of source data
PASSTHROUGHCopy data without any modification
SUBSETCopy a filtered or sampled subset of rows
SKIPExclude the table from processing entirely

Data Mapping Actions

ActionDescription
MASKReplace the column value using the configured masking strategy
MIGRATE_AS_ISCopy the original value to the destination unchanged

Masking Strategies

CLI Usage

# Authenticate
opendatamask-cli auth login --url http://localhost:8080 \
  --username admin --password secret

# Workspaces
opendatamask-cli workspace list
opendatamask-cli workspace get <workspace-id>

# Jobs
opendatamask-cli job list --workspace <workspace-id>
opendatamask-cli job run  --workspace <workspace-id>

Privacy & Compliance

The Privacy Hub dashboard shows:

  • Total sensitive columns detected vs. masked
  • Actionable recommendations (e.g., "Add EMAIL generator to users.email")
  • Exportable JSON compliance reports for GDPR / CCPA / HIPAA audits

Run Apply Recommendations to automatically assign generators to all unmasked sensitive columns in one click.

Troubleshooting

SymptomFix
JWT_SECRET must be setSet the JWT_SECRET environment variable before starting
Backend fails to startEnsure PostgreSQL is running and DATABASE_URL is correct
Connection refused on port 8080Run docker-compose logs backend to check for errors
Sensitivity scan finds nothingAdd table configurations first, then re-run the scan
Job fails with "no source connection"Ensure at least one connection is marked as a Source role in the workspace
Screenshots not generatedRun pip install playwright && playwright install chromium then re-run take_screenshots.py with the frontend running
